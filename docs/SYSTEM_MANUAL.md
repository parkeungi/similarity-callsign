## 1. 시스템 개요

### 1.1 목적
항공사 운항 중 발생하는 **유사 호출부호(Similar Callsign)** 상황을 감지하고, 항공사별 조치를 추적·관리하는 폐쇄형 웹 서비스입니다.

### 1.2 사용자 구분

| 역할 | 설명 | 접근 범위 |
|------|------|-----------|
| **관리자 (Admin)** | 항공교통본부 담당자 | 전체 시스템 관리, 데이터 업로드, 사용자 관리 |
| **항공사 사용자 (User)** | 각 항공사 담당자 | 자사 유사호출부호 조회, 조치 등록/관리 |

### 1.3 핵심 업무 흐름

```
[관리자] 엑셀 업로드 → 유사호출부호 데이터 등록
    ↓
[항공사 사용자] 발생현황 확인 → 조치 등록
    ↓
[시스템] 양사 조치 상태 자동 동기화
    ↓
[관리자] 전체 조치현황 모니터링 → 통계 확인
```

## 3. 인증 및 로그인

### 3.1 로그인 화면 (`/`)

**화면 구성**:
- 시스템 로고 및 명칭
- 이메일 입력 필드
- 비밀번호 입력 필드
- 로그인 버튼

**로그인 처리 흐름**:

```
1. 사용자가 이메일/비밀번호 입력 후 [로그인] 클릭
2. POST /api/auth/login 호출
3. 서버에서 users 테이블 조회 → bcrypt 비밀번호 검증
4. 성공 시 JWT Access Token 발급 (payload: userId, email, role, airlineId)
5. 클라이언트: sessionStorage에 토큰 저장, Zustand authStore 업데이트
6. 역할별 자동 리다이렉트:
   - 관리자 → /callsign-management?tab=overview
   - 항공사 사용자 → /airline
```

**에러 처리**:

| 상황 | 응답 |
|------|------|
| 이메일 미존재 | 401: "이메일 또는 비밀번호가 올바르지 않습니다." |
| 비밀번호 불일치 | 401: "이메일 또는 비밀번호가 올바르지 않습니다." |
| 계정 정지 | 403: "계정이 정지되었습니다." |

### 3.2 헤더 (공통)

로그인 후 모든 페이지 상단에 표시되는 공통 헤더입니다.

**좌측 영역**:
- 태극마크 + "국토교통부 항공교통본부" 텍스트
- "유사호출부호 경고시스템" 시스템명 (클릭 시 홈으로 이동)
- 소속 항공사명 (또는 관리자: "항공교통본부")

**우측 하단 영역**:
- 로그인 사용자 이메일 표시
- 관리자 전용: [유사호출부호] [관리자 페이지] 링크
- [Logout] 버튼

**로그아웃 처리**:
```
1. [Logout] 클릭
2. POST /api/auth/logout (쿠키 삭제)
3. Zustand store 초기화, sessionStorage 클리어
4. /login 페이지로 리다이렉트
```

┌─────────────────────────────────────────────────────────┐

##### ③-b. 발생 건 카드 목록 (중앙)

각 유사호출부호 발생 건이 카드 형태로 표시됩니다 (기본 9건/페이지).

**카드 1건의 구조 (좌→우, 위→아래)**:
```
┌──────────────────────────────────────────────┐
│  KAL852 | AAR851          [매우높음] [유사:높음]│  ← 호출부호쌍 + 위험도/유사도 배지
│  오류유형: 관제사오류 > 복창오류                  │  ← 오류유형 + 세부오류
│  발생: 3회  |  최초: 2026-01-15               │  ← 발생횟수 + 최초발생일
│  출발: ICN  |  도착: NRT                      │  ← 출발/도착 공항 (있는 경우)
│                                              │
│           [조치상태: ⏳ 미조치]  [조치등록 →]     │  ← 상태 배지 + 조치등록 버튼
└──────────────────────────────────────────────┘
```

| 요소 | 설명 |
|------|------|
| 호출부호 쌍 | "자사코드+편명 \| 타사코드+편명" 형태 |
| 위험도 배지 | 🔴 매우높음 (빨강) / 🟠 높음 (주황) / 🟢 낮음 (초록) |
| 유사도 배지 | 높음 / 중간 / 낮음 |
| 오류유형 | 관제사오류 / 조종사오류 / 오류미발생 + 세부오류 |
| 발생횟수 | 누적 발생 건수 |
| 조치상태 | ⏳ 미조치 / 🔄 조치중 / ✅ 완료 |

**카드 클릭 동작**:
| 동작 | 결과 |
|------|------|
| **카드 본문 클릭** | → 발생 상세 모달 표시 (⑤-c 참고) |
| **[조치등록] 버튼 클릭** | → 조치등록 모달 표시 (⑤-a 참고) |

| 3 | **조치유형** | 드롭다운 (관리자가 등록한 조치유형 목록에서 선택) |

**등록 기술 흐름**:
```
[등록] 클릭
→ POST /api/airlines/{airlineId}/actions
   Body: { callsign_id, action_type, description, planned_due_date, completed_at, status }
→ 서버: actions 테이블 INSERT + callsigns 상태 동기화
→ 성공: modal.closeModal() → queryClient.invalidateQueries(['airline-actions'])
→ 목록 자동 갱신
```

**완료 조건 매트릭스** (callsigns 상태 자동 판정):

| 항공사 조합 | 조치 주체 | callsigns 최종 상태 |
|------------|----------|-------------------|
| 같은 항공사 (자사↔자사) | 아무나 | completed |
| 국내 ↔ 국내 | 첫 번째 항공사만 | in_progress |
| 국내 ↔ 국내 | 두 번째 항공사도 | completed (양사 완료) |
| 국내 ↔ 외항사 | 국내 항공사 | completed |
| 외항사 ↔ 외항사 | 아무나 | completed |

> **국내항공사 판별**: airlines 테이블에 등록된 11개사 (KAL, AAR, JJA, JNA, TWB, ABL, ASV, EOK, FGW, APZ, ESR)만 국내항공사.

#### 탭 1: 발생현황 (Overview)
- 항공사 페이지의 발생현황과 유사하나, **전체 항공사**의 데이터를 조회
- 추가 필터: 항공사 선택 드롭다운
- 각 건별로 **양사의 조치 상태**를 함께 표시

#### 탭 2: 조치현황
- 호출부호 쌍별로 양쪽 항공사의 조치 진행 상태를 한눈에 확인
- 상태 표시: ✅ 완료 (양사 모두) / 🔄 부분 완료 (한쪽만) / ⏳ 미조치

#### 탭 3: 통계
- 시스템 전체 통계 차트
- 위험도별 분포, 항공사별 조치 완료율, 월별 탐지 추이

#### 탭 4: 엑셀입력

**위→아래 구조**:

| 순서 | 영역 | 내용 |
|------|------|------|
| 1 | **파일 업로드 영역** | 드래그 앤 드롭 또는 [파일 선택] 버튼 |
| 2 | **업로드 결과** | 전체 행 수, 성공 건수, 실패 건수 + 사유 |
| 3 | **업로드 이력** | 파일명, 업로드일, 업로드자, 성공/실패 건수 |

**업로드 기술 흐름**:
```
[파일 선택 또는 드래그 앤 드롭]
→ POST /api/admin/upload-callsigns (multipart/form-data)
→ 서버: Excel 파싱 → 유효성 검증 → callsigns 테이블 INSERT
→ file_uploads 테이블에 이력 기록
→ 결과 JSON 반환 → 화면에 성공/실패 건수 표시
```

## 9. 보안 및 인증 체계

### 9.1 JWT 토큰 구조

| 항목 | AccessToken | RefreshToken |
|------|------------|--------------|
| 알고리즘 | HS256 | HS256 |
| 서명 Secret | `JWT_SECRET` | `REFRESH_TOKEN_SECRET` (별도 분리) |
| audience | `katc1:access` | `katc1:refresh` |
| issuer | `katc1` | `katc1` |
| 만료 | 1시간 | 7일 |
| 저장 | sessionStorage | httpOnly 쿠키 |
| DB 저장 | 없음 | SHA-256 해시 저장 (`users.refresh_token_hash`) |

**토큰 혼용 공격 차단**: audience/issuer 검증으로 RefreshToken을 AccessToken 자리에 사용하는 공격 차단

### 9.2 RefreshToken 무효화 흐름

```
로그인:
  → RefreshToken 생성 → SHA-256 해시 → DB 저장 + httpOnly 쿠키 설정

토큰 갱신 (/api/auth/refresh):
  → 쿠키의 RefreshToken JWT 서명 검증
  → 해시 계산 → DB 저장값과 비교 (불일치 = 탈취 토큰 → 401)
  → 통과 시: 새 토큰 생성 + 새 해시 DB 저장 (이전 토큰 즉시 무효)

로그아웃:
  → DB refresh_token_hash = NULL (탈취된 토큰도 즉시 무효화)
  → 쿠키 삭제
```

### 9.3 로그인 보안 (행안부 정보보호 지침)

| 정책 | 설정 |
|------|------|
| IP Rate Limiting | 10회/분 초과 시 429 응답 |
| 계정 잠금 | 5회 연속 실패 → 15분 잠금 |
| 잠금 자동 해제 | 15분 경과 후 자동 해제 |
| 비밀번호 만료 | 90일마다 변경 강제 |
| 비밀번호 이력 | 최근 5개 재사용 방지 |
| 비밀번호 복잡도 | 8자 이상, 대/소문자+숫자+특수문자 |
| 비밀번호 찾기 | 이메일 발송 없음 → 관리자 문의 안내 |

관련 DB 컬럼: `users.failed_login_attempts`, `users.locked_until`

### 9.4 API 보호

```
모든 API 요청:
  → Authorization: Bearer {accessToken} 헤더 필수
  → 서버에서 JWT 검증 (audience, issuer, 서명 모두 확인) → 실패 시 401
  → 관리자 API: role === 'admin' 추가 검증 → 실패 시 403
```

### 9.5 SQL Injection 방지

```
모든 DB 쿼리는 파라미터화 쿼리 사용:
  ✅ query('SELECT * FROM users WHERE id = ?', [userId])
  ❌ query(`SELECT * FROM users WHERE id = '${userId}'`)

DB 관리 API: 테이블명 화이트리스트 검증 (ALLOWED_TABLES Set)
```

### 9.6 입력값 검증

| 검증 대상 | 규칙 |
|-----------|------|
| 이메일 | 유효한 이메일 형식 |
| 비밀번호 | 8자 이상, 대/소문자+숫자+특수문자 (`PASSWORD_REGEX`) |
| 위험도 | "매우높음", "높음", "낮음" 만 허용 |
| 상태값 | 정의된 enum 값만 허용 |
| DB 테이블명 | ALLOWED_TABLES 화이트리스트 |

### 9.7 필수 환경변수 (보안)

| 변수명 | 용도 | 생성 방법 |
|--------|------|-----------|
| `JWT_SECRET` | AccessToken 서명 | `openssl rand -base64 32` |
